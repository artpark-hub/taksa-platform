# Modbus Facade API — UI Developer Guide

Typed endpoints under `/api/v1/devicemgmt/devices/{device_id}/protocol-converters/modbus` build and parse umh-core payloads server-side. Use these for the Modbus configuration UI. Generic `/protocol-converters` remains for power users.

## Endpoints

| Action | Method | Path | Returns |
|--------|--------|------|---------|
| Deploy | `POST` | `.../protocol-converters/modbus` | `actionId` (workflow) |
| Edit | `PATCH` | `.../protocol-converters/modbus/{uuid}` | `actionId` (single action) |
| Get config | `GET` | `.../protocol-converters/modbus/{uuid}` | `actionId` (single action) |
| Poll typed result | `GET` | `.../protocol-converters/modbus/actions/{action_id}/result` | status + optional `result` |
| List inventory | `GET` | `.../protocol-converters` | catalog rows (`type`, health, deployment) |
| Delete | `DELETE` | `.../protocol-converters/{uuid}` | `actionId` |

OpenAPI: `device-management/openapi.yaml` (search `Modbus`).

## Async flow (required)

All deploy / edit / get calls are **async**. Poll until `status` is `COMPLETED` or `FAILED`.

```
Deploy:  POST modbus → poll workflow actionId → optional GET for verify
Edit:    PATCH modbus/{uuid} → poll actionId → GET modbus/{uuid} (mandatory for fresh config)
Get:     GET modbus/{uuid} → poll actionId → result is full facade
```

Poll every 1–2s. Use `expiresAt` as a soft timeout hint.

## Poll semantics (important)

| Source | `COMPLETED` + `result` |
|--------|-------------------------|
| **GET** action | Full `ModbusProtocolConverter` |
| **Deploy workflow** | Full facade when configure succeeded (from applied config if device reply is minimal) |
| **Edit** action | Often **no `result`** — device returns `{uuid}` only. Treat `COMPLETED` as success; **run GET** for config. |

Never assume edit poll returns YAML. Always refresh with GET before showing the form again.

## Response shape: `ModbusProtocolConverter`

Dual-mode sections — same object for display and edit:

- `input` — Modbus-specific `mode` + `structured` + `rawYaml` (controller, slave IDs, unified/legacy addresses, advanced tuning)
- `readFlow` — shared with OPC-UA facade (`processorMode`, `bufferMode`, tag processor, conditions)
- `rawInputYaml`, `rawProcessorYaml`, `rawBufferYaml` — mirrors for RAW editors
- `inputParse`, `processorParse` — `PARSE_OK` | `PARSE_PARTIAL` | `PARSE_FAILED`
- `deploymentStatus`, `healthStatus`, `errorMessage` — from catalog (list + GET poll)
- `protocol` (`modbus`), `processing_mode`, `state`

`output` is read-only (UNS autogenerated on device).

## Section modes (edit / deploy)

When both structured and raw are present, set mode explicitly:

| Field | Values |
|-------|--------|
| `input.mode` | `STRUCTURED` \| `RAW` |
| `readFlow.processorMode` | `STRUCTURED` \| `RAW` |
| `readFlow.bufferMode` | `STRUCTURED` \| `RAW` |

**STRUCTURED** — server builds YAML from structured fields.  
**RAW** — server uses `rawYaml` / `rawProcessorYaml` / `rawBufferYaml` verbatim.

Omitting mode when both structured and raw are set → `400`.

### Modbus input (structured)

- **Standard**: `controller` (`tcp://{{ .IP }}:{{ .PORT }}`), `slaveIds`, `unifiedAddresses` (preferred), legacy `addresses`, `timeBetweenReads`, `timeout`
- **Advanced**: `transmissionMode`, `optimization`, `byteOrder`, `busyRetries`, `workarounds`, etc.
- **Unified address format**: `name.register.address.type[:key=value]*` (see taksa-benthos-umh `modbus` input docs)

### Read flow (shared tag processor)

Reuse the OPC-UA `readFlow` shape. Conditions reference **`modbus_tag_name`** (not `nodeId`):

```json
"conditions": [
  {
    "joiner": "AND",
    "clauses": [
      {
        "field": "modbus_tag_name",
        "operator": "equals (===)",
        "value": "FirstFlagOfDiscreteInput"
      }
    ],
    "action": "msg.payload = Number(msg.payload) > 0 ? 1 : 0;\nreturn msg;"
  }
]
```

Use **RAW** processor mode when the device has YAML-list conditions (`- if:` blocks) not yet editable in structured form.

## Deploy request

**JSON field names use camelCase** (same as OPC-UA facade, Topic Browser, and other Device Management JSON APIs). Request and response use the same names (`readFlow`, `applyReadConfig`, `templateVariables`, etc.).

```json
{
  "deviceId": "...",
  "name": "line-1-modbus",
  "state": "active",
  "applyReadConfig": true,
  "connection": { "ip": "192.168.1.10", "port": 502 },
  "location": { "levels": { "0": "Plant", "1": "Line1" } },
  "input": {
    "mode": "STRUCTURED",
    "structured": {
      "standard": {
        "controller": "tcp://{{ .IP }}:{{ .PORT }}",
        "slaveIds": [1],
        "unifiedAddresses": [
          "FirstFlagOfDiscreteInput.holding.1025.UINT16"
        ],
        "timeBetweenReads": "1s",
        "timeout": "1s"
      }
    }
  },
  "readFlow": {
    "processorMode": "STRUCTURED",
    "bufferMode": "STRUCTURED",
    "processor": {
      "defaults": {
        "locationPath": "{{ .location_path }}",
        "dataContract": "_historian"
      }
    },
    "yamlInject": { "rawYaml": "buffer:\n  none: {}\n" }
  },
  "templateVariables": { "IP": "192.168.1.10", "PORT": "502" }
}
```

- `name` is required; **uuid is derived from name** (deterministic, same as umh-core).
- `applyReadConfig: false` → shell only (connection/location), no read pipeline.
- Poll **workflow** `actionId`, not child deploy/configure IDs.

## Edit request

Same fields as deploy plus `uuid`. Typed edit is **full replace** of read config — send complete `input` + `readFlow` every time.

After `COMPLETED`, run GET to reload the form.

## GET as source of truth

Use GET poll `result` to populate the UI. Structured fields are best-effort parse of device YAML:

- Legacy `addresses` and `unifiedAddresses` are both parsed when present.
- `PARSE_PARTIAL` — complex YAML may only appear in `rawInputYaml` or `rawProcessorYaml`; keep RAW mode available.

## List vs typed GET

| Field | List | Typed GET |
|-------|------|-----------|
| `type` | `modbus` (set at facade deploy) or `protocol-converter` for generic/legacy bridges | `protocol` on GET result refines if needed |
| Health / deployment | yes | yes (merged from catalog on poll) |
| Full YAML / structured | no | yes |

Filter list by `type=modbus` to show only Modbus bridges.

## Errors

| HTTP | Meaning |
|------|---------|
| `400` | Validation (missing connection, ambiguous modes, build error) |
| `404` | Unknown action / device |
| `409` | Deploy: name already exists |
| `412` FailedPrecondition | Typed GET/EDIT on non-Modbus bridge (catalog type mismatch) |
